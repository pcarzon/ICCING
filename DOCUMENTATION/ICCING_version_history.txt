Version History for ICCING code

_____________v0.0.1_____________

(Finished 5-31-2019)

-	Basic functionality established
-	Modular code with individual routines and a master routine which calls the others
-	Generates own MC Glauber event with tuneable parameters
-	Discretizes event into discrete numbers of gluons in each cell
-	Reads in pre-computed quark splitting ratios from file
-	Scans the grid and probabilistically samples the outcome of gluon splittings
-	Redistributes energy and charge to new grid cells after splitting
-	Computes eccentricities of entropy and individual quark flavors (u, ubar, d, dbar, s, sbar, c, cbar)
-	Writes output to file in a format which can be read directly into v-USPhydro

_____________v0.0.2_____________

(Finished 6-11-2019)

-	Adapted to read events generated by TRENTO as input, rather than using own MC Glauber event generator

_____________v0.1.0_____________

(Finished 6-14-2019)

-	Built a new gluon sampling / redistribution algorithm which gives a spatial extent to the gluons and to the produced quarks
-	Extensive testing to ensure energy and charge conservation are maintained throughout the evolution
-	Algorithm designed and tested to ensure it does not warp the shape of the initial density profile too much
-	Mass thresholds implemented for various quark flavors (rainbow cake effect)
-	Samples random gluon energy distribution, but uses constant probability distribution rather than a physical (1/x) distribution
-	Fixed an error in the radial probability distribution (previously omitted phase space density factor r)  [Thanks, Mauricio!]

_____________v0.1.1_____________

(Finished 6-14-2019)

-	Included the MV model for the dipole amplitude as an option
-	Fixed an error in the radial probability distribution (previously omitted Gev*fm conversion factors) [Thanks, Mauricio!]
-	Modified the eccentricity calculation to include the conserved charges WRT the energy CMS

_____________v0.1.2_____________

(Finished 6-21-2019)

-	Added a subroutine for generating high-quality plots of output
-	Fixed a bug where the improved charge redistribution algorithm was calling the discretized, artificially-inflated "energy" distribution
-	Added the initialization time to obtain 3D density distributions instead of 2D

_____________v0.1.3_____________

(Finished 6-22-2019)

-	Added a routine to read in the equation of state and use it to convert from T_R in TRENTO to actual 3D energy density
-	Modified the ViewTrento routine to display the energy density plot with absolute scale and total energy deposited
-	Modified the charge redistribution algorithm to use the 3D energy density correctly (previously assumed a 2D energy density)
-	Improved the efficiency of the redistribution algorithm:
	>	Only sampling gluon energies up to the available energy in the selected region
	>	Switching to incrementing / decrementing the total energy rather than recomputing it from the grid
-	Added a truncation energy Echop which rounds extremely small energy densities ~10^{-70} to zero

_____________v0.1.4_____________

(Finished 6-23-2019)

-	Added a global debugging flag which prints out intermediate data useful for debugging and cross-checks.
-	Fixed an error in the q qbar distance (r = total q qbar distance, vs. r = displacement from gluon position)
-	Added a movie-making subroutine

_____________v0.1.5_____________

(Finished 7-1-2019)

-	Changed eccentricity routines to calculate the minimum number of quantities
-	Changed output routines to only output the eccentricities from an event, not the event itself, and add events line by line to a shared file
-	Added global variables "eventnum" and "Stot" read from TRENTO to keep track of the event number and total entropy
-	Removed old "Stot" information which was taken from adding up the T_R or energy grids, not the Stot directly from Trento
-	Small changes to progress update messages


_____________v0.1.6_____________

(Finished 7-2-2019)

-	New routine to select positive vs negative regions of the charge grids
-	New routine to make "mesons" and "baryons" from various flavors by taking geometric means of the corresponding valence quarks
-	Modified eccentricity routines to compute \epsilon_{1,3} estimator instead of \epsilon_1 which is identically zero (except for conserved quantities which sum to zero)
-	Added calculation of eccentricities of positive vs negative B/S/Q
-	Fixed a small issue with the check which avoids trying to calculate eccentricities for an empty matrix, which caused it to not work with negative-definite matrices


_____________v0.2.0_____________

(Finished 7-5-2019)

-	Major reconfiguration of the code to be run on the Amarel cluster.  The code is now split into 3 parts:
	>	ICCING_input.wl		(input file containing run parameters)
	>	ICCING_v0_2_0.wl	(body of the algorithm; defines all functions)
	>	ICCING_wrapper.wl	(can be run from a command line with Mathematica on it; reads the input and calls the core routine to process 1 event)

-	There are also 3 associated script files which are written in the Unix Bash shell script to call the various routines on the cluster:
	>	bscript.sh		(for a given "job" number, loops the "wrapper" file to run many events -- currently 25 at a time)
	>	bslurm.sh		(submits many "jobs" in parallel to the cluster by calling bscript.sh multiple times on different cores)		
	>	bsubmit.sh		(called from command line; takes arguments to modify parameters in bslurm.sh; also creates a macro to monitor events as they run)

-	Various features of the .sh files are unique to the environment of the Amarel cluster: modules being loaded, Matt's directory structure, etc.

-	The body of the code, ICCING_v0_2_0, has not been changed from v0.1.6 other than to convert it to a .wl file which is able to be run from the command line

-	Most of the run parameters are kept in ICCING_input.wl, but certain directory settings have been moved to the .sh files
	>	This helps set up the directory structure "/output/JOBLABEL/JOBNUMBER/" which then contains events numbered 0 - 24 (at present)
	>	JOBLABEL is a string which briefly describes the conditions of the job: "GBW," "MV_0241", etc.
	>	JOBNUMBER generally starts at 0 and can be increased to provide the desired number of events.  To obtain 10K events for a given job, this goes from 0 - 399.

-	ICCING_wrapper.wl should be able to be called from command line on any machine with Mathematica, assuming that it can find ICCING_input.wl and ICCING_v0_2_0.wl
	>	The syntax to run it is "wolfram -script ICCING_wrapper.wl LABEL JOBNUMBER EVENTNUMBER"
		>>	Looks in the folder "~/ICCING/output/LABEL/" for ICCING_input.wl
		>>	Looks in the folder "~/ICCING/" for ICCING_v0_2_0.wl
		>>	Reads Trento input files like "ic0.dat", "TA0.dat", "TB0.dat" and writes eccentricity output files "LABEL_jobJOBNUMBER_evEVENTNUMBER_Ecc.dat" 
			in the folder "~/ICCING/output/LABEL/JOBNUMBER"

-	The shell script bscript.sh should run on a single machine with Unix.  
	>	Intended to first generate Trento events, but Trento itself is not working on the cluster.  
		>>	For now, copies over 25 pre-generated events to run on.
		>>	Will try to get Trento running again on the cluster ASAP to generate new events.
	>	Runs ICCING_wrapper.wl on a given event number from 0-24, producing the output file "blahblah_Ecc.dat" and then deletes the corresponding "ic", "TA", and "TB" files
	

-	The shell script bslurm.sh contains many commands that are specific to the "Slurm" language for job control on a computing cluster.  
	>	The syntax to run it is "sbatch bslurm.sh LABEL"
	>	All the #SBATCH commands set important parameters which are fed to the job manager on the cluster
	>	The parameters which will change every time are the job name and array values for the job ID number.  These are left as variables which are replaced by bsubmit.sh
	>	Creates appropriate directory substructure in "~/ICCING/output/" for the job and events.
	>	Copies the run parameter files (from ICCING: ICCING_input.wl, and from Trento: eventsetup.conf) to the new subdirectory for reference on the conditions of the job
	>	Submits individual jobs to run bscript.sh to available cores on the cluster
		>>	The output from running bscript.sh (which includes all the progress messages generated by Mathematica) is saved as the file term_JOBNUMBER.dat

-	The shell script bsubmit.sh reads in parameters for the job submission to the cluster, and submits them by modifying bslurm.sh
	>	The syntax to run it is ". ./bsubmit.sh LABEL JOBSTART JOBEND"
		>>	The punctuation at the beginning is important: ". ./bsubmit.sh"
		>>	LABEL is the label for the job folder; also used for the job name on the cluster and the cluster error and output files
		>>	JOBSTART and JOBEND are the numerical values for the jobs being submitted to the cluster, e.g. 0 and 399 to send 400 jobs at once
	>	Takes the generic parameters MYLABEL, MYSTART, MYEND as written in bslurm.sh and replaces them with the values provided as LABEL, JOBSTART, end JOBEND
		>>	Actually makes a temporary copy of bslurm.sh called bmod.sh after making the replacements
	>	Submits the jobs by calling sbatch on bmod.sh
	>	Uses the job parameters to define a macro (called an "alias") which prints out summary information about the jobs as they run, so you can check on their progress.
	>	Deletes the temporary file bmod.sh

-	The macro which checks on the progress of the jobs is named "ck" (standing for "check").  It is created every time bsubmit.sh is run, according to the job parameters
	>	Prints out the status of the first and last job in the batch submitted to the cluster, which includes showing their runtimes
	>	Prints out the contents of the job folders for the first and last events in the batch
		>>	You can watch as the input files "ic", "TA", and "TB" get converted to "blahblah_Ecc.dat" as individual events finish
	>	Prints out the last 5 lines of the "term_JOBNUMBER.dat" files which show the most recent Mathematica progress messages
		>>	You can tell if the job is working on calculating something ("Redistributing quarks" or "Calculating eccentricities") or if it's frozen in some intermediate step
		>>	If you run "ck" right after submitting the job, the "term_0.dat" files won't have 5 lines to display, so they may clear the screen
			or look weird until after a few seconds have passed and the first event is underway.

_____________v0.1.7_____________

(Finished 2-24-2020)

-	Obtained from modifications of v0.1.6 (GUI version, not cluster version)
-	Changed equation of state reading to allow for reading Jan's new equation of state
-	Added new run parameters
	>	EOSInputFile	Name of EOS file to be read
	>	EOSOmitLines	Arguments of Drop[...] to indicate which irrelevant lines from EOS file need to be dropped before being used
	>	EOSScol		Column number of EOS file corresponding to the entropy density
	>	EOSEcol		Column number of EOS file corresponding to the energy density
-	ConvertTrento routine changed to read from arbitrary EOS file
	>	Also changed the construction of output SparseArray for energy grid; old routine allowed for accidental mapping of "0" entropy to nonzero energy following the EOS.  Now this is explicitly avoided.
-	Realized that axes were being rotated in plots.  Actual (x,y) values are converted to array (i,j) indices by i = (x - xmin)/dx and y = (y - ymin)/dy so that (xmin,ymin) coincides with (0,0).
-	MatrixPlots of matrix M_{i j} need to be transposed first to obtain M_{j i} and then DataReversed to obtain M_{jmax -  j, i} so that axes are oriented properly
	>	Unfortunately, doing this also reversed the labels on the Y-axis, so now it looks like it's being plotted from (+12 fm) to (-12 fm)
	>	{x,y} coordinates in output file are correct, however...
-	Fixed an incorrect label on the full HydroInput file which is generated; previously the 3rd column was labeled "s", it has now been fixed to "E" (energy density)
-	Similarly, the sum of all grid points was being reported as "Stot"; it is now reported as "Etot" (total energy) and correctly multiplies by dx * dy * tau0 to get total energy in GeV
-	Changed the graphic printout to display the final (modified) energy density rather than the original (clean) energy density.
-	Fixed the grid points being written to file.  Previously they were incorrectly divided by the grid area (dx * dy).  After fixing this, the raw grid 
	points are output, which were checked to verify that the integrated energy (sum over grid * dx * dy * tau0) agrees with the initial tabulated energy.


_____________v0.1.8_____________

(Finished 7-1-2020)

-	Working on adding energy dependence of gluon quanta.  Incorporated a flexible form for user's choice of power law, verified the MC reproduces curves.
-	Discovered a mistake in the MC sampling of the splitting variables \alpha, r:  the MC ceiling used is incorrect and way too low, leading to an incorrect sampling of the curve.
-	Redid the MC ceiling using a root-finder at \alpha = 0 to fix the absolute ceiling.  
-	Moved the \alpha sampling into the MC routine (instead of first choosing \alpha from a uniform distribution and then choosing r from the MC).  
	Now the multivariate probability distribution is being sampled simultaneously in both variables.
-	Did extensive testing to verify that the new MC does correctly reproduce the multivariate input distribution.
	Validation was successful.  Tested multiple quark masses and multiple Qs values.  Still works when the peak is greater than r_max.
